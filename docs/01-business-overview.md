# 01. Business Overview — 한양유화 e-Business OS

## 1. 회사 컨텍스트

한양유화는 폴리에틸렌(PE) 및 석유화학 원료를 유통하는 한화솔루션 대리점이다. 한화솔루션 제품 외에 타사 국산 원료, 수입 원료(SGCG, QAPCO, Reliance, Formosa 등)도 함께 취급한다.

### 현재 운영 도구

- **클릭2002** — 한화솔루션 대리점 전용 주문/출고 시스템 (Windows 데스크탑 클라이언트)
- **이카운트 ERP** — 매출/재고/회계 관리 (웹 + 데스크탑)
- **공유 스프레드시트 오더시트** — 영업사원이 주문을 입력
- **Python 자동화 스크립트 8개** — 사무실 데스크탑 PC에서 실행, 오더시트 → 클릭2002/이카운트 자동 입력
- **카카오톡 / 전화 / 이메일 / 문자** — 거래처 주문 접수 채널 (체계화되지 않음)

### 현재 운영의 9가지 문제점

1. 영업사원이 오더시트에 입력하지 않으면 주문이 누락된다.
2. 전화·카톡·문자·이메일 주문이 체계적으로 관리되지 않는다.
3. 주문 접수 후 담당자가 처리하지 않으면 상태가 방치된다.
4. 보류 건이 잊혀진다.
5. 승인된 주문도 배차 실패로 당일 출고 안 됨.
6. 출고 후 실제 출고량과 전산 입력 수량이 다를 수 있다.
7. 매일 17:00 ERP 입력 작업이 사람 손에 의존한다.
8. 상태 변경 시 거래처/내부 크로스체크가 없다.
9. 대표/관리자 입장에서 한눈에 보는 경영 지표가 없다.

## 2. 본 시스템의 목표

> 한양유화의 **주문 접수 → 담당자 확인 → 승인/반려/보류 → 공급처 주문 → 배차 → 출고 → ERP 입력 → 거래처 알림 → 마감 → 경영지표 분석**까지 연결하는 내부 주문 관제 시스템 및 향후 고객 주문 포털의 기반을 만든다.

단순 홈페이지 주문 기능이 아니다. **사내 주문 관제 시스템**이 1차 목표이다.

## 3. 기존 마케팅 사이트와의 관계

| 도메인                   | 역할                             | 코드베이스                             | 변경          |
| ------------------------ | -------------------------------- | -------------------------------------- | ------------- |
| `www.hanyangpetro.com`   | 회사소개, 제품자료, 물성표, SEO  | Hugo 정적 사이트 (`hanyangpetro` 리포) | **변경 없음** |
| `order.hanyangpetro.com` | 내부 주문 관제 + (2차) 고객 포털 | Next.js 동적 앱 (본 리포)              | 신규          |

DNS에 CNAME 1줄(`order` → Cloudflare Tunnel)만 추가하면 된다. 도메인 추가 구매·기존 사이트 수정 모두 불필요. 자세한 설정은 [`09-hosting-and-domain.md`](./09-hosting-and-domain.md) 참조.

## 4. 사용자 역할

| 역할                     | 코드                     | 권한 요약                                                                              |
| ------------------------ | ------------------------ | -------------------------------------------------------------------------------------- |
| 대표/임원                | `EXECUTIVE`              | 전사 조회, 경영 대시보드, 예외 승인, 여신/미수 위험 조회, ERP 실패 조회                |
| 관리팀/영업지원/전산입력 | `ADMIN` (또는 `SUPPORT`) | 전체 주문 조회/수정, ERP 입력 후보 검토·실행, 입력 실패 수동 처리, 마스터 데이터 관리  |
| 영업담당자               | `SALES`                  | 본인 담당 거래처 주문 조회·등록, 승인/반려/보류, 보류 재알림 설정, 배차/출고 상태 입력 |
| 거래처 담당자 (2차 MVP)  | `CustomerUser`           | 본인 회사 주문요청·상태 확인·재주문, 출고 예정 확인, (선택) 수령 확인                  |

영업담당자/관리자는 **모든 거래처와 모든 도착지를 검색·입력**할 수 있다. 거래처 사용자는 **본인 회사 데이터만** 보고, 본인 회사에 등록된 도착지(필수 입력)와 화이트리스트된 제품(과거 출고 이력 있는 것)만 선택할 수 있다 — 주문 실수 방지가 목적.

## 5. MVP 범위 결정

### 1차 MVP (이번 작업)
- 내부 직원 관제 화면 우선
- mock 알림 발송 (실제 이메일/카톡 미연동)
- ERP 입력은 후보 생성 + 관리팀 승인 + (mock) 실행 + 결과 기록 구조까지
- 거래처 포털: DB 모델·라우팅만 준비, UI는 미구현

### 2차 MVP (이후)
- 거래처 로그인 + 주문 포털
- 화이트리스트 기반 제품 노출
- 실제 카카오 알림톡/SMS 연동 (유료, 의사결정 필요)
- 클릭2002/이카운트 자동입력 어댑터 실연동 (기존 Python 8개 통합)
- AI 기반 주문 원문 파싱 (전화/카톡 메모를 구조화 데이터로 변환)
- 자동 수령 간주 정책 (거래처가 2영업일 무응답 시) — 대표 승인 필요

## 6. 완료 기준 (1차 MVP)

본 README의 "완료 기준" 10개 항목 (요구사항 §20):
- ✅ 주문 상태 흐름 정의
- ✅ 승인/반려/보류/보류 재알림 구조
- ✅ 배차 중/완료/실패/재시도 구조
- ✅ 출고 후 실제 수량 차이 가능성 반영
- ✅ 거래처 수령 확인 optional 구조
- ✅ 17:00 ERP 입력 후보 생성 프로세스
- ✅ 이카운트/클릭2002 자동입력 batch 구조 (어댑터)
- ✅ 거래처/담당자 알림 로그 구조
- ✅ 대표/관리자 대시보드 기초
- ✅ 기존 홈페이지와 분리되어 안전 개발
